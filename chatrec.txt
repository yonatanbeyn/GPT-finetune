  Critical Issue: Memory

  Full fine-tuning of Qwen3-4B requires ~32GB VRAM (FP16). This is likely not feasible on a consumer GPU.

  You must use LoRA (PEFT) instead of full fine-tuning:

  from peft import get_peft_model, LoraConfig, TaskType

  lora_config = LoraConfig(
      task_type=TaskType.CAUSAL_LM,
      r=16,
      lora_alpha=32,
      target_modules=["q_proj", "v_proj"],
      lora_dropout=0.05,
  )
  model = get_peft_model(model, lora_config)
  # Trainable params drop from ~4B to ~10M

  With LoRA + 4-bit quantization (load_in_4bit=True via bitsandbytes), you can run on ~6-8GB VRAM.

   The Problem: bitsandbytes doesn't support MPS

   load_in_8bit=True uses the bitsandbytes library — CUDA only. On your M2, it will crash.

   NotImplementedError: bitsandbytes is not supported on MPS

   ---
   Your Options on M2

   Option 1 — MLX (Apple-native, recommended)
   Apple's own ML framework, optimized for M-series chips, supports native quantization:
   pip install mlx mlx-lm
   mlx_lm.lora --model Qwen/Qwen3-4B --train --data data/ --bits 8
   This is the cleanest path for M2 fine-tuning.

   Option 2 — llama.cpp / GGUF Q8
   Convert Qwen3-4B to GGUF Q8 format and fine-tune with llama.cpp — works on MPS natively.

   Option 3 — If you have a CUDA GPU available
   Then bitsandbytes INT8 works fine:
   from transformers import BitsAndBytesConfig

   bnb_config = BitsAndBytesConfig(load_in_8bit=True)
   model = AutoModelForCausalLM.from_pretrained(
       "Qwen/Qwen3-4B",
       quantization_config=bnb_config,
       device_map="auto",
   )
   Then layer LoRA on top of that — trainable params stay in FP16, frozen layers in INT8.

     2. "Writing model shards" — What Is Sharding?

     Writing model shards: 1/1 [00:00<00:00, 1.50it/s]

     Sharding = splitting a model's weights across multiple files.

     - GPT-2 (124M params) fits in 1 shard → 1/1 (one file: model.safetensors, 475MB)
     - Qwen3-4B would split into multiple shards e.g. 1/4, 2/4... because the weights exceed the single-file limit (~5GB)
     - The HuggingFace Trainer does this automatically via safetensors

     With Qwen3-4B + LoRA you'd see:
     Writing model shards: 1/4 ...
     Writing model shards: 2/4 ...
     And a model.safetensors.index.json index file pointing to all shards.

     ---
     3. Warning: missing keys ['lm_head.weight']

     There were missing keys in the checkpoint model loaded: ['lm_head.weight']

     This is not a real error. It happens because:
     - GPT-2 uses weight tying — lm_head.weight shares the same tensor as transformer.wte.weight (the embedding matrix)
     - When saving, only one copy is stored
     - When reloading, HuggingFace warns that lm_head.weight isn't a separate key — but it's correctly reconstructed from the tied embedding

     With Qwen3-4B this warning goes away as it handles weight tying differently.

     Harmless — Qwen3's config.json includes a loss_type field that older Trainer versions don't recognize. It falls back to standard causal LM loss, which is

  What the model learns from the data:
