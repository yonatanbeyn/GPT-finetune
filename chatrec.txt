     2. "Writing model shards" — What Is Sharding?

     Writing model shards: 1/1 [00:00<00:00, 1.50it/s]

     Sharding = splitting a model's weights across multiple files.

     - GPT-2 (124M params) fits in 1 shard → 1/1 (one file: model.safetensors, 475MB)
     - Qwen3-4B would split into multiple shards e.g. 1/4, 2/4... because the weights exceed the single-file limit (~5GB)
     - The HuggingFace Trainer does this automatically via safetensors

     With Qwen3-4B + LoRA you'd see:
     Writing model shards: 1/4 ...
     Writing model shards: 2/4 ...
     And a model.safetensors.index.json index file pointing to all shards.

     3. Warning: missing keys ['lm_head.weight']

     There were missing keys in the checkpoint model loaded: ['lm_head.weight']

     This is not a real error. It happens because:
     - GPT-2 uses weight tying — lm_head.weight shares the same tensor as transformer.wte.weight (the embedding matrix)
     - When saving, only one copy is stored
     - When reloading, HuggingFace warns that lm_head.weight isn't a separate key — but it's correctly reconstructed from the tied embedding

     With Qwen3-4B this warning goes away as it handles weight tying differently.

● Great question. The answer is: there's no actual "deciding to think" — it's pure next-token prediction.

  Here's what happens during fine-tuning:

  What the model learns from the data:
  Q: [question]\n<think>\n[reasoning]\n</think>\nA: [answer]
  The model sees this pattern 79 times and learns: "after a Q: block, the high-probability next tokens are <think>".

  During inference:
  - You give it Q: What is logit?\nA: — but wait, you're skipping <think>, so the model is already off-pattern.
  - If you instead give it Q: What is logit?\n, the model predicts <think> next simply because that token had high conditional probability in training.

  How to run

  Prereqs on Windows 11:
  1. Docker Desktop with the WSL2 backend enabled (Settings → General → "Use WSL 2 based engine").

  docker compose build

  Checkpoints land in GPT-finetune/output/ on your host (mounted volume).

  Inference (interactive):
  docker compose run --rm finetune python inference.py

  Single-prompt inference:
  docker compose run --rm finetune python inference.py "Machine learning is"

  Without compose (plain docker):
  docker build -t gpt-finetune .
  docker run --rm --gpus all -v ${PWD}/output:/app/output gpt-finetune

  Key design notes

  - Base image is pytorch/pytorch:2.4.0-cuda12.1-cudnn9-runtime — torch + CUDA are prebuilt, so the build is fast and there's no driver/toolkit version dance.
  - deploy.resources.reservations.devices in compose is the modern way to request GPUs (replaces the old runtime: nvidia).
  - output/ is bind-mounted so checkpoints survive container removal; hf-cache is a named volume so gpt2 weights don't re-download each run.

    1 # Containerizing GPT-finetune — Architecture & GPU Passthrough
    2
    3 This doc explains **what was added**, **how the host GPU reaches the container**, and **why this is the same model SageMaker uses on EC2**.
    4
    5 ---
    6
    7 ## 1. What changed
    8
    9 Before containerization, training ran directly on the host Python interpreter — it touched the OS pip env, the host CUDA toolkit, and wrote checkpoints into the project tree.
   10
   11 After containerization, everything except the GPU driver lives inside an immutable image. The host now provides three things only: the kernel, the NVIDIA driver, and a directory to mount for c
      heckpoints.
   12
   13 | Concern               | Before                    | After                                                    |
   14 | --------------------- | ------------------------- | -------------------------------------------------------- |
   15 | Python + torch + CUDA | Installed on host         | Baked into `pytorch/pytorch:2.4.0-cuda12.1-cudnn9` image |
   16 | Dependencies          | `pip install` on host     | Resolved once at `docker build`                          |
   17 | GPU access            | Direct (host process)     | Injected via NVIDIA Container Toolkit                    |
   18 | Checkpoints (`output/`) | Written in project tree | Bind-mounted to host directory                           |
   19 | HF model cache        | `~/.cache/huggingface`    | Named Docker volume `hf-cache` (survives rebuilds)       |
   20 | Reproducibility       | Whatever's on the machine | Image digest = exact env, anywhere                       |
   21
   22 ### Files added
   23 - `Dockerfile` — image recipe (base + deps + source).
   24 - `.dockerignore` — keeps `.git/`, `output/`, `__pycache__/` out of the build context.
   25 - `docker-compose.yml` — declares the GPU reservation, volumes, and run command.
   26
   27 ### One pin that matters
   28 The base image ships **torch 2.4.0**. The default `transformers` on pip is now 5.x and registers a `torch.library.custom_op` whose tensor signature 2.4 can't parse. The Dockerfile pins `transf
      ormers>=4.40,<5` to dodge this. If you upgrade the base image to torch 2.5+, you can drop the pin.
   29
   30 ---
   31
   32 ## 2. How CUDA reaches the container
   33
   34 The container does **not** ship a GPU driver. It ships the CUDA *runtime* (`libcudart`, cuDNN, the PyTorch wheels). The *driver* (`libcuda.so`, kernel module) stays on the host. The NVIDIA Con
      tainer Toolkit splices the two together at container start.
   35
   36 ### Layered view
   37
   38 ```
   39 +----------------------------------------------------------+
   40 |  Container (gpt-finetune image)                          |
   41 |    - python 3.11, torch 2.4 (CUDA 12.1 runtime), HF      |
   42 |    - finetune.py / inference.py                          |
   43 |    - calls torch.cuda.is_available()  ──┐                |
   44 +-----------------------------------------│----------------+
   45                                           │ dlopen("libcuda.so")
   46                                           ▼
   47 +----------------------------------------------------------+
   48 |  /usr/lib/x86_64-linux-gnu/libcuda.so                    |
   49 |  ── BIND-MOUNTED from host at container start ──         |
   50 +-----------------------------------------│----------------+
   51                                           │ ioctl(/dev/nvidia*)
   52                                           ▼
   53 +----------------------------------------------------------+
   54 |  WSL2 kernel  → /dev/nvidia0, /dev/nvidiactl, /dev/nvidia-uvm  |
   55 |  (exposed to WSL by the host driver, no kernel module in WSL) |
   56 +-----------------------------------------│----------------+
   57                                           │
   58                                           ▼
   59 +----------------------------------------------------------+
   60 |  Windows NVIDIA driver 581.83  ──►  RTX 4060 (physical)  |
   61 +----------------------------------------------------------+
   62 ```
   63
   64 ### The handshake, step by step
   65
   66 1. **You ask for GPUs.** In `docker-compose.yml`:
   67    ```yaml
   68    deploy:
   69      resources:
   70        reservations:
   71          devices:
   72            - driver: nvidia
   73              count: all
   74              capabilities: [gpu]
   75    ```
   76    (Same as `docker run --gpus all` on the CLI.)
   77
   78 2. **Docker Desktop sees that request and invokes `nvidia-container-runtime`** instead of plain `runc`. The runtime is shipped with Docker Desktop on Windows — no separate install.
   79
   80 3. **Before the container's `init` runs**, the runtime executes an OCI **prestart hook** (`nvidia-container-cli`). It does three things:
   81    - Bind-mounts the host's `libcuda.so` and matching userspace libs into the container.
   82    - Bind-mounts `/dev/nvidia0`, `/dev/nvidiactl`, `/dev/nvidia-uvm` so the process can `ioctl()` the driver.
   83    - Adjusts cgroups so the container actually has permission to use those devices.
   84
   85 4. **PyTorch starts.** `import torch; torch.cuda.is_available()` succeeds because:
   86    - `libcudart.so.12` (runtime, in the image) finds `libcuda.so` (driver, mounted from host).
   87    - The driver version (581.83 → CUDA 13.0 capable) is ≥ the runtime's required version (12.1). **This direction matters** — host driver must be ≥ container's CUDA runtime. Newer host, older
      container = fine.
   88
   89 5. **`finetune.py:27 get_device()` returns `"cuda"`** and training runs on the host GPU. We confirmed this end-to-end: training ran to completion in 21:30 on the RTX 4060, with ~4 GB of GPU me
      mory in use on the host.
   90
   91 ### Why the container can be "thin"
   92 The CUDA runtime is ~1 GB; the driver is hundreds of MB. By keeping the driver on the host and only mounting it in, the toolkit avoids:
   93 - Shipping driver binaries per-image (would need re-release every driver update).
   94 - Driver/kernel-module version skew (an old container would refuse a new kernel).
   95
   96 This is the same reason you can run a CUDA 11 container on a CUDA 13 host driver, but **not** the reverse.
   97

❯ how inference.py is used

● Here's how inference.py works and the ways to invoke it.

  What it does

  It loads the fine-tuned model from config.OUTPUT_DIR (output/), wraps each prompt in the same Q:/<think>/A: template the model was trained on, generates up to MAX_NEW_TOKENS=150 tokens, and splits
  the output into the thinking part (between <think> and </think>) and the answer part (after A:).

  Key spots in the file:
  - inference.py:22-27 — picks mps / cuda / cpu. Inside the container it picks cuda.
  - inference.py:33-40 — loads tokenizer + model from output/, calls .eval(), moves to device.
  - inference.py:46-81 — wraps the prompt as Q: {prompt}\n<think>\n, calls model.generate(...) with the sampling params from config.py, then splits on </think> and stops at \nQ: or \nPrompt>.
  - inference.py:98-106 — single-shot mode: if you pass the prompt as a CLI arg, it generates once and exits.
  - inference.py:108-125 — interactive mode: with no args, it sits in a REPL printing Prompt>  until you type quit / exit or hit Ctrl-C.

  Generation defaults (override in config.py:32-36):

  What python inference.py "..." is actually doing

  When you run:
  docker compose run --rm finetune python inference.py "What is deep learning?"

  docker compose sees you passed extra args (python inference.py "...") and uses them to override the image's default CMD (which is ["python", "finetune.py"]). The shell inside the container then runs
  python inference.py "What is deep learning?" — and inference.py is found at /app/inference.py (the WORKDIR).

  So your CLI is selecting which baked-in script to run, not uploading anything.

  The consequence to know about

  If you edit inference.py on the host, the running container won't see the change. You have two options:

  1. Rebuild (cheap because layer caching skips pip):

    Why PyTorch alone is 2.6 GB

    libtorch_cuda.so ships precompiled GPU kernels for every NVIDIA architecture PyTorch supports: sm_50, sm_60, sm_70, sm_75, sm_80, sm_86, sm_89 (your 4060), sm_90 (H100). That's 8 copies of every op
    (matmul, conv, softmax, …). It's the price of "pip install torch and it works on any GPU."

    cuDNN adds another ~1 GB because it bundles precomputed conv/attention plans — lookup tables of optimal kernel parameters for common tensor shapes. That's what makes the first iteration fast.

    What you can trim — realistic options

    1. Switch base to nvidia/cuda:12.1.1-runtime-ubuntu22.04 + pip-install torch directly.
    Skips the conda overhead. Saves ~1–2 GB.
    FROM nvidia/cuda:12.1.1-runtime-ubuntu22.04
    RUN apt-get update && apt-get install -y python3.11 python3-pip && rm -rf /var/lib/apt/lists/*
    RUN pip install torch --index-url https://download.pytorch.org/whl/cu121
    2. Use the *-runtime-rocm or -slim variants of the PyTorch image if/when they exist for your torch version.
    3. Multi-stage build to drop pip cache & wheels. With PIP_NO_CACHE_DIR=1 already set, you've got this benefit. Saves ~100–300 MB.
    4. Strip unused architectures from PyTorch. Theoretical — requires rebuilding PyTorch from source. Could save ~700 MB. Not worth it.
    5. Remove torchvision + torchaudio if you truly don't need them:
    RUN pip uninstall -y torchvision torchaudio
